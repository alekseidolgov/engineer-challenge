# Процесс разработки с использованием AI

Инструмент: **Cursor IDE** (Claude).
Роль AI: генерация кода по архитектурным указаниям.
Роль инженера: архитектурные решения, code review, корректировки, security audit.

## Хронология решений

### 1. Scope и стек

**AI предложил:** Go + React, gRPC-Gateway для HTTP.
**Моё решение:** убрать фронтенд, оставить чистый gRPC. Фокус на архитектурной зрелости backend, а не на вёрстке.

### 2. Email валидация

AI упростил валидацию до `mail.ParseAddress`. Я спросил — зачем дополнительная проверка на точку в домене? AI объяснил: `mail.ParseAddress` пропускает `user@localhost`. Оставили проверку, добавили комментарий на русском и ограничение по длине (254 символа по RFC 5321).

### 3. Игнорирование ошибки в user_repo.go

Нашёл на code review: `email, _ := domain.NewEmail(emailStr)` — ошибка молча проглатывалась. Указал AI на строку 70, исправлено на полноценную обработку.

### 4. Безопасность хеша Argon2id

Спросил: зачем хранить параметры хеширования внутри хеш-строки — это же упрощает задачу атакующему? AI объяснил: формат PHC стандартен, параметры не секрет, а необходимость — позволяют верифицировать старые хеши после смены параметров без массового сброса паролей. Убедительно — оставил, добавили комментарий.

### 5. JWT: HS256 → ES256

AI по умолчанию использовал HS256. Я задал вопрос — почему не HS512? AI объяснил trade-off. Затем я уточнил: у нас планируется multi-service архитектура. AI предложил ES256 (асимметричная подпись) — приватный ключ только у auth-сервиса, публичный раздаётся потребителям. Принял. Также договорились описать миграцию на ES512 как next step.

### 6. Rate limiting: in-memory → Redis

AI реализовал in-memory rate limiter. Я спросил: это нормальное решение для distributed среды? Нет. Перевели на Redis с Sliding Window Counter (Lua-скрипт). Я спросил — какой алгоритм, почему именно этот, что отклонили? AI описал Fixed Window (burst на стыке), Token Bucket (burst by design), Sliding Window Log (O(n) память). Sliding Window Counter — оптимальный баланс.

### 7. Миграции: entrypoint SQL → golang-migrate → отдельный контейнер

AI изначально монтировал SQL через Docker entrypoint. Я возразил: нет версионирования, идемпотентности. Перешли на `golang-migrate` с `go:embed`. Затем я спросил: если сервисов 10, на каждом запускать миграции? Нет — вынесли в отдельный контейнер `migrator` (в k8s — Job/init-container). Также зафиксировали next step: вынести мигратор в shared-модуль.

### 8. Bounded Context: два → один

AI выделил два bounded context: `IdentityAccess` и `CredentialRecovery`. Я спросил — почему два, если recovery мутирует тот же User aggregate и использует те же password policy? AI согласился: один контекст `Identity` с recovery как поддоменом. README и ADR обновлены с обоснованием по критериям Эванса.

### 9. Kubernetes манифесты

Я спросил: можем сделать Terraform/K8s? AI предложил варианты. Выбрал K8s манифесты с Kustomize — проверяемо через `kubectl --dry-run`, практичнее Terraform без реального облака. Затем я спросил: смогу ли проверить на minikube? AI добавил `postgres.yaml` и `redis.yaml` для локального тестирования.

### 10. Security audit

Попросил проверить код на уязвимости, в частности SQL-инъекции. AI провёл аудит. Результат:
- **SQL-инъекции: 0** (все запросы параметризованные)
- **Найдено и исправлено:**
  - Password без max length → DoS через Argon2id (добавлен лимит 72)
  - Reset token без max length → DoS через SHA-256 (добавлен лимит 128)
  - Error enumeration: ошибки reset token раскрывали состояние (unified в generic message)
  - User ID в ошибке corrupt email (убран)

## Итог

AI генерировал ~90% кода. Мои корректировки затронули:
- Выбор архитектуры (pure gRPC, один bounded context, ES256, Redis rate limiter)
- Инфраструктурные решения (отдельный мигратор, K8s вместо Terraform)
- Code review (проглоченная ошибка, отсутствие input limits, error enumeration)
- Понимание trade-offs на каждом шаге
